# Дипломна робота: опис реалізації TransitMind

**Тема:** «Програмне забезпечення для системи картографічної платформи з візуалізацією даних».

Цей документ зібрано як **єдине джерело правди** про архітектуру, функціонал, зовнішні API, візуалізації та обмеження проєкту — для розділів записки, презентації або передачі в LLM (Gemini тощо).

**Мова UI застосунку:** українська та англійська (`frontend/src/i18n/messages.js`).

---

## 1. Відповідність темі диплому

| Аспект теми | Реалізація в TransitMind |
|-------------|--------------------------|
| **Картографічна платформа** | Інтерактивна карта на **Leaflet** (базовий шар Esri World Dark Gray, перемикач супутника), відображення міст, ребер графа, маршруту (полілінія), маркерів зупинок, симуляції руху під час сесії. |
| **Програмне забезпечення** | Клієнт **React + Vite**, сервер **Node.js + Express**, БД **PostgreSQL**, маршрутизація по дорожній мережі через **OSRM** (публічний інстанс). |
| **Візуалізація даних** | Бічна панель з метриками маршруту; **SVG sparkline** температури вздовж траєкторії; порівняльні **смуги** часу поїздки (TomTom free-flow / historic / live); **стовпчикова діаграма + лінія тренду** тривалості поїздки за різних годин виїзду «завтра»; списки погодних карток; підказка «навантаження» з середньої швидкості OSRM. |

---

## 2. Мета та призначення системи

**TransitMind** — вебзастосунок для планування маршруту на карті (Україна як основний контекст даних), перегляду **відстані та часу** поїздки (OSRM), отримання **контексту середовища** (погода Open-Meteo, часовий пояс), опційно — **трафік та затримки** (TomTom Routing API за наявності ключа), запуску **локальної подорожі** у вигляді сесії з анімацією руху по траєкторії та таймером. Система спрощена: **один локальний користувач**, без соціальних функцій, чатів і обов’язкового логіну.

---

## 3. Архітектура (логічна)

```
[Браузер: React SPA]
    │  HTTPS (axios → /api)
    ▼
[Backend: Express]
    ├── PostgreSQL (маршрути, сесії, граф міст/доріг)
    ├── OSRM (router.project-osrm.org)
    ├── Open-Meteo
    ├── timeapi.io (timezone)
    └── TomTom Routing API (опційно)
```

Сервер виступає **агрегатором**: ключ TomTom не потрапляє у клієнт; зовнішні виклики з бекенду.

## 5. Користувацькі сценарії

1. **`/`** — головна, перехід до маршрутів.
2. **`/routes`** — вибір точок, «Деталі маршруту» (OSRM dry run), панель даних, «Старт» (сесія в БД + перехід).
3. **`/session/:id`** — карта, таймер, інтерполяція позиції по полілінії; leave завершує сесію.

---

## 6. Зовнішні джерела даних

| Джерело | Дані |
|---------|------|
| OSRM | Маршрут driving, відстань (м), тривалість (с), координати |
| Open-Meteo | Погода по точках |
| timeapi.io | Часовий пояс |
| TomTom | Час у дорозі, затримка, порівняння сценаріїв, профіль виїздів |

**Чесність формулювань:** «навантаження» без TomTom — евристика з **середньої швидкості OSRM**, не стрімінг і не дані інших користувачів TransitMind.

## 8. REST API (скорочено)

Префікс клієнта: `.../api` (див. `frontend/src/services/api.js`).

- `GET /health`, `GET /api/health`
- `GET/POST` під `/api/routes`, `/api/sessions`
- `GET /api/dijkstra/cities`, `GET /api/dijkstra/roads`
- `POST /api/dijkstra/find-route`, `find-route-multi`, **`find-route-osrm`**
- `GET /api/dijkstra/route-line`
- `POST /api/context/route-points`, **`/api/context/route-enrichment`**

Тіло `route-enrichment`: `pathCoordinates`, `statistics` → відповідь з `weatherAlongRoute`, `routePace`, `liveTraffic`, `congestionByDeparture`, `sources`.

## 9. Алгоритми

1. **Dijkstra** на графі з БД (`backend/src/algorithms/dijkstra.js`).
2. **OSRM multi-stop** — зшивання сегментів (`dijkstra.js`).
3. **Інтерполяція** вздовж полілінії за прогресом сесії (`Session.jsx`).

## 12. Обмеження

- Залежність від публічного OSRM і квот TomTom.
- Погода — по семплах «зараз», без повного прогнозу по часу прибуття в кожну точку.
- Немає реального GPS — симуляція за таймером сесії.

---

## 13. Перспективи

Кеш enrichment, експорт звітів, додаткові постачальники трафіку, розділення режимів «граф Dijkstra» vs «OSRM» у UX.

---

## 14. Мета / об’єкт / предмет (чернетка для записки)

- **Мета:** ПЗ вебплатформи з побудовою маршруту, відображенням на карті та графічним поданням метеорологічних і транспортно-часових атрибутів.
- **Об’єкт:** візуалізація просторових і тематичних даних у web GIS-подібному застосунку.
- **Предмет:** інтеграція зовнішніх геосервісів і засоби графічної подачі результатів.
